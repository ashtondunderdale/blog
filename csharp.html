Write code for humans first

NOVEMBER 27, 2023 | 4 min read ASHTON DUNDERDALE


If you have been coding for a while, you have probably heard the phrase 'premature optimisation': the act of changing how a program functions to make it faster before it is necessary. Developers often encounter this common pitfall, often justifying it solely based on performance concerns but never considering its priority level over readability.

It looks something like this:

"You know, if you replace straightforward array iteration with a complex bitwise operation, it will marginally boost the performance of your book sorting algorithm." 🤓

The statement is correct, but the person is arguably wrong. It will increase the performance as bitwise operations typically execute in one clock cycle whereas array processing involves traversing stored elements, introducing additional latency, but computers are very fast and the impact it will have on real-world performances are negligible.


Take the 'for loop' for instance, and its counterpart, the 'foreach' (C#). For loops will iterate until a certain condition is true. They have three parts: an initial declaration statement, a conditional statement that iterates until false, and a statement that will be executed each iteration. Foreach loops are different in that it simply iterates through the collection, for the amount of items in the collection. The key difference being that instead of using an indexed element, it uses a loop variable to perform the iteration.

For:

int[] numbers = { 1, 2, 3, 4, 5 };

for (int i = 0; i < numbers.Length; i++)
{
    int squared = numbers[i] * numbers[i];
    Console.WriteLine($"The square of {numbers[i]} is {squared}");
}

Foreach:


int[] numbers = { 1, 2, 3, 4, 5 };

foreach (int number in numbers)
{
    int squared = num * num;
    Console.WriteLine($"The square of {num} is {squared}");
}

Noticeably, readability is greatly enhanced by the foreach loop versus the for. So much so that the syntax of the foreach loop is comparable to that of plain english: "for each number in numbers". Not to mention that, depending on the program you are writing, 99.9% of your program's total runtime will likely be waiting for user input, that said, micro optimisation should not be applicable in that scenario (or most for that matter).



So, which is it?

While the provided example above shows how you will almost always use the foreach loop for simple array iteration, take note that it is case-specific. In scenarios where explicit control over the loop index is necessary, such as iterating through elements conditionally or in reverse order, a for loop would be more suitable.


This form of micro-optimisation can be seen almost everywhere in software development.

Imagine this: Barry is a software developer that is spending a significant amount of time and resources implementing complex functionalities and cutting-edge technologies, anticipating that these features will make him stand out in the market. But, unfortunately he did not ensure the basic features of his product met the needs and expectations of the user base. Nobody cared that his groundbreaking, state-of-the-art data analytics engine could generate intricate cash spending plans because his product did not allow users to log their income and expenses. Barry's company went bankrupt.

To be an efficient developer, it is good practice to begin coding with a focus on readability, reserving performance optimisation interventions for cases where it is measured and deemed required. We should prioritise code that not only functions correctly but is also comprehensible. Readability takes precedence over raw speed and performance if it compromises clarity.


In the words of Knuth (Hoare): Premature optimisation is the root of all evil.
